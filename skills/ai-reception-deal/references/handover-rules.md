# handover-rules

本文件定义 AI 接待何时必须转人工，以及转人工时必须输出哪些信息。

## 1. 必须转人工的场景

### 1.1 已有强烈成交意向

- AI 已收集到客户完整需求，且客户提出发合同、PI 或其他成交文件要求
- 客户要求最终成交价、紧急正式报价或超出标准范围的特殊条件
- 客户已进入深度议价，且具备明显成交意向

### 1.2 高价值客户

- 客户涉及多产品 / 多 SKU / 多配置 / 多规格，或明显的大批量采购

### 1.3 有特殊要求客户

- 客户提出特殊付款方式、特殊账期、责任承诺、赔偿条款或法务问题
- 客户对政策、出口、赔偿或责任问题持续深入追问
- 客户提出投诉、争议或要求明确责任归属
- 当前问题需要管理判断，而知识库无法提供足够支撑

### 1.4 KB 无目标 SKU 且 qualify 已齐

跟 SKILL.md §3.2 "承接优先于 KB 状态披露" 配套——前者管"qualify 没齐时怎么继续承接"，本条管"qualify 齐了之后怎么交接"。

触发条件（同时满足）：

- `lookup_product` 对客户提到的 SKU 返回 `not_found`
- 当前 product_line 的 `qualify_fields` 已经全部补齐到 leads

行为要求：

- `route = HUMAN_NOW`
- `handoff_summary` 必须包含：
  - 客户原始 SKU 表述（保留客户原话，便于销售判断是否能调配 / 推近似品）
  - 已收集的 qualify 字段值
  - 标签 `"待确认 SKU"`
- `next_message` **不暴露 KB miss**——不出现 "我们没有 X" / "暂未收录" / "目录里没有" 这类措辞。参考表达：
  > "您的需求我们已经记下，稍后我们这边同事会和您具体确认 [SKU] 的安排"

## 2. 转人工时必须输出的内容

- 客户当前诉求
- 已确认的产品和交易信息
- 已完成的回复内容
- 当前所处谈单阶段
- 尚未补齐的关键信息
- 触发人工接手的原因

## 3. 转人工的表达原则

- 对客户：平稳、自然、不中断体验
- 对内部：信息完整、可直接接手
- 不要只说“我帮你转人工”，必须有明确交接摘要
