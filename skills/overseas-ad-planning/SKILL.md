# 汽车出海 Meta CTW 广告投放 SOP

## 1. 运行环境硬约束(必须遵守)

本 skill 运行在 LeadEngine 宿主环境内,**不是** Claude.ai 通用环境。以下能力**不存在**,任何提及它们的指令都会失败:

- ❌ 没有文件系统:不能"保存到 outputs/"、不能 `present_files`、不能 `Write`/`Edit`
- ❌ 没有 Python 沙箱:不能"运行下面的脚本"
- ❌ 没有 OpenRouter / Meta API 直连:不能让用户"自备 API Key"
- ❌ 没有 Bash / Shell:不能"运行命令"

### 1.1 输出形式硬规则

- **所有产出物**(市场分析、广告策略、素材清单、Meta 方案)以**完整 markdown 内容**直接出现在 assistant 回复里
- **禁止**说"已保存到 xxx.md,请查收附件"——用户看不到任何附件
- **禁止**说"详细内容因篇幅省略"——必须输出完整内容
- 长内容用标准三反引号包裹

### 1.2 工具白名单(只有 5 个)

| 工具 | 调用时机 |
|---|---|
| `web_search` | §4.C 规则查询、阶段二取最新数据、阶段 1.5 决策辅助 |
| `read_webpage` | 配合 web_search 深读单一来源 |
| `read_skill_reference` | 按需读取 references/*.md(详见各阶段说明) |
| `generate_ad_creative` | **仅阶段四**生成广告图,每素材调一次 |
| `draft_ad_plan` | **由宿主 host-patch 负责调用**,skill 内文不写 |

**禁止**调用任何其他工具(file I/O、shell、Python、第三方 API 直连等都不存在)。

## 2. 产品锁定:Meta CTW(Click-to-WhatsApp)

本 skill 只支持一种广告形式:**Meta Click-to-WhatsApp**(用户点击广告 → 跳转 WhatsApp 对话)。

### 2.1 锁定字段(plan_json 中必须严格使用,不允许其他枚举)

| Meta 字段 | 锁定值 |
|---|---|
| `campaign.objective` | `OUTCOME_ENGAGEMENT` |
| `campaign.special_ad_categories` | `[]` |
| `campaign.buying_type` | `AUCTION` |
| `campaign.campaign_budget_optimization` | `true` |
| `adset.optimization_goal` | `CONVERSATIONS` |
| `adset.billing_event` | `IMPRESSIONS` |
| `adset.destination_type` | `WHATSAPP` |
| `adset.promoted_object` | `{ page_id, whatsapp_phone_number }` |
| `adset.targeting.targeting_automation` | `{ advantage_audience: 0 }` |
| `creative.link_data.link` | `https://api.whatsapp.com/send` |
| `creative.link_data.call_to_action.type` | `WHATSAPP_MESSAGE` |
| `creative.link_data.call_to_action.value.app_destination` | `WHATSAPP` |
| `campaigns.length`(数组长度) | **严格 = 1**(多市场用同 campaign 下多个 ad_sets 按 `targeting.countries` 切分) |

### 2.2 禁用清单(不要在任何阶段提及作为可选项)

- **禁用 objective**:`OUTCOME_LEADS` / `OUTCOME_TRAFFIC` / `OUTCOME_AWARENESS` / `OUTCOME_SALES` / `OUTCOME_APP_PROMOTION`
- **禁用 optimization_goal**:`LEAD_GENERATION` / `LANDING_PAGE_VIEWS` / `LINK_CLICKS` / `OFFSITE_CONVERSIONS` / `IMPRESSIONS` / `REACH`
- **禁用 CTA type**:`LEARN_MORE` / `GET_QUOTE` / `SIGN_UP` / `CONTACT_US` / `SHOP_NOW` / `SUBSCRIBE` / `BOOK_TRAVEL` / `WATCH_MORE`
- **禁用**:任何 Lead Form 配置、任何指向落地页的 link、Pixel 落地页转化事件、CRO 章节

### 2.3 不在范围内

| 项 | 状态 |
|---|---|
| 视频素材 | 不支持(generate_ad_creative 仅出 1:1 静态图,需扩展时由宿主升级) |
| 多平台(Google / LinkedIn / TikTok) | 不支持(走对应平台 skill) |
| Lead Generation / Traffic 等非 CTW 投放 | 不支持(由宿主整体设计决定) |
| 投放后数据回流分析 | 由独立 skill 处理 |
| 多 campaign | 锁定为 1 个 |

## 3. 阶段框架(独立调用 + 依赖校验 + 阶段间锁定)

```
阶段 1.0:需求收集 + 业务模式识别
    ↕
阶段 1.5:决策辅助(仅在用户字段不全时触发)
    ↕
§4.C:平台 + 国家规则查询(1.0+1.5 完成后自动执行,一次性锁定)
    ↕
阶段 2:市场分析(数据告诉用户「市场怎么样,优势在哪,往哪投」)
    ↕                                依赖关系(自上而下递进,修改只往后传)
阶段 3:广告策划/策略(告诉用户「想怎么投」)
    ↕
阶段 4:CTW 素材生成(图片 + 文案 + welcome_message 文本)
    ↕
阶段 5:执行方案(全部 Meta 参数集中输出 plan_json + 后台操作手册)
```

### 3.1 阶段独立调用规则

阶段**不强制顺序执行**,允许按用户场景从任意阶段进入。但阶段间有**依赖关系**——每阶段开始前按下表「输入契约」校验前置产出:

| 当前阶段 | 输入契约(必须有) | 缺失时处理 |
|---|---|---|
| 阶段 1.0 | 用户对话或 brief 文件 | 直接进入,启动收集 |
| 阶段 1.5 | 阶段 1.0 产出中存在「待确认/帮我推荐」标记的字段 | 无此标记则跳过 1.5 直接执行 §4.C |
| §4.C 规则查询 | 阶段 1.0(+ 1.5) 已确定目标平台 + 目标国家 | 1.0+1.5 全部完成后自动触发,一次性查询并锁定 |
| 阶段 2 | 阶段 1.0(+ 1.5)产出 + §4.C 规则锁定结果 | 缺失任一必填字段 → 先调阶段 1.0(快速模式) |
| 阶段 3 | 阶段 1.0(+ 1.5)产出 + 阶段 2 的市场分析结论 + §4.C 规则锁定 | 缺失则先补对应阶段(快速模式可现场调研补足关键结论) |
| 阶段 4 | 阶段 1.0(+ 1.5)产出 + 阶段 3 的策略要点 + §4.C 规则锁定 + **用户已上传产品图** | 无产品图必须先提示上传;策略要点缺失则先补阶段 3 |
| 阶段 5 | 阶段 3 产出 + 阶段 4 产出(**含真实素材 url**) + §4.C 规则锁定 | 缺失则先补对应阶段 |

### 3.2 缺失处理两档

- **缺失项较少**(1-3 个字段):直接对话提问补全,然后继续当前阶段
- **缺失项较多**(>3 个字段或整个前置阶段缺失):走「快速通道」,用最简输入启动该阶段,所有缺失字段标 `[待客户补充]` 继续输出。**不要回退到前置阶段强制重跑**

### 3.3 sanity check 仍然生效(关键)

宿主有反 shortcut 保护:走到 `draft_ad_plan` **仍必须有**真实 `generate_ad_creative` 调用记录(同会话内已有过即可,多轮迭代场景成立)。`plan_json` 中 image_url 必须来自真实工具调用,不能伪造。

**实际行为**:阶段五开头校验输入契约时,如果发现"无 generate_ad_creative 调用记录"且"用户没提供已有 image_url",必须**拦截并提示用户先跑阶段四**,不要硬出 plan_json。

### 3.4 必须保留的机制

- 阶段间**依赖关系不变**(虽然不强制顺序执行,但依赖链如上表)
- 每阶段产出后**简要复述核心要点并询问用户是否需要调整**,确认后再继续
- 阶段五完成后**不要写"流程到此结束"**等收尾语;宿主会接续调用 `draft_ad_plan` 完成 CTW 收口提交

### 3.5 阶段间结论锁定 + 修改交互流程

#### 3.5.1 阶段间锁定(内部维持,不展示版本号给用户)

- AI **内部**为每阶段产出维护一个 `latest_conclusion` 字段(纯内部使用,不展示)
- 每阶段结束时,AI 内部更新 `latest_conclusion`,并以「简短描述段落」给用户复核(几行 bullet,不是大段复述)
- 下一阶段开始时,AI **内部**读 `latest_conclusion` 作为输入,不再依赖对话历史搜索
- 版本号机制纯内部使用,**永远不展示给用户**(用户看到的永远是最终版)

**简短描述段落示例**(展示给用户的样子,3-5 行 bullet):

```markdown
阶段二结论速览:
• 目标市场:菲律宾(主)+ 哈萨克斯坦(备)
• 推荐打法:面向商用车队的 Cost-Saving 调性
• 关键合规预警:不写具体补贴金额、不写价格绝对值
• 数据时效:已基于 2026 年 4 月最新报告
```

#### 3.5.2 修改方向规则(关键)

- **用户修改只往后传,不回改前置阶段**(例:用户在阶段三说「市场改越南」,assistant 只更新阶段三及之后,不改阶段二)
- **例外**:用户明确说「请回去改阶段 N」,assistant 才回改,且声明「后续阶段需要重做」并征求用户确认

#### 3.5.3 修改交互流程

按「用户提出的修改点数量」分两种处理:

| 修改点数量 | AI 处理方式 |
|---|---|
| **1-2 个修改点** | AI 识别修改点 + 同章节关联部分 → 逐个修改 → 直接输出整段更新版给用户(不需要中间确认) |
| **≥3 个修改点** | AI 识别全部修改点 + 同章节关联部分 → 逐个修改,每改完一处给用户看修改的部分 → 全部改完后询问「已完成 N 处修改,是否输出完整版?」→ 用户确认后输出整段更新版 |

**「关联部分」判断标准**:仅限「**同阶段同章节内**」的关联,不跨阶段。举例:用户在阶段三第 04 章把「目标市场→越南」,assistant 同步改同阶段 05 章受众关键词(语言/文化变了)、10 章多语言文案(他加禄语→越南语),**不**回去改阶段二的市场分析。

## 4. 阶段一(拆成 1.0 需求收集 + 1.5 决策辅助 + 4.C 规则查询)

### 4.A 阶段 1.0:需求收集

#### 4.A.1 用户角色识别(对话开场第一问)

| 选项 | 后续差异 |
|---|---|
| 品牌方(车厂/经销商) | 标准产出 + 简明决策记录段落 |
| 代理商(4A/数字代理) | 标准产出 + 提案大纲(SCQA 结构) |
| 第三方出海平台 | 标准产出 + 多品牌批量化建议段落 |

#### 4.A.2 业务模式识别(开场第二步)

**识别流程**:

1. AI 从用户上传的需求里**自动识别**业务模式(关键词:跨境批发/经销商对接/海外零售/品牌官方等)
2. 如果用户描述清晰 → AI 内部记录识别结果,直接进入字段收集
3. 如果用户描述不清晰 → AI 简短问一句:「您想推这个车给谁?他们的用车场景是什么?」让用户用一两句话描述
4. AI 根据描述**自定义**本次任务的:漏斗、受众词、文案语气、KPI 指标

**业务模式单向禁忌词典**:

| 模式倾向 | 单向禁忌词(严禁出现) | 风格倾向(优先使用) |
|---|---|---|
| **B2C 终端零售** | FOB / MOQ / OEM / CBU / 集装箱 / 报关 / 货代(术语错位,消费者听不懂) | 试驾 / 到店 / 家用 / 续航焦虑 / 智能化 |
| **B2B 跨境批发** | (无严禁词,试驾在大客户场景可出现) | MOQ / FOB / 经销商 / OEM 支持 / 售后保障 / 渠道资源 |
| **品牌官方混合** | 无 | 综合,根据用户描述自定义 |
| **自定义模式** | AI 从用户描述抽取 | AI 从用户描述抽取 |

「风格倾向」是优先建议,不是硬限制。详细模板调用 `read_skill_reference({ name: "strategy-template" })` 第 02 章。

#### 4.A.3 必填字段(8 项,缺一不启动后续阶段;对话式收集,不一次抛全部问题)

1. **品牌**(中英文)
2. **车型**(中英文) — 若用户标注「待确认/请帮我推荐」→ 触发阶段 1.5
3. **车辆类型**(纯电 SUV / 混动轿车 / 燃油性能车 / 家用 MPV / 商用皮卡 / 轻卡 / 等)
4. **核心卖点**(3 个,客户原话)
5. **目标国家**(具体 ISO-2 国家代码;多国时按预算占比说明) — 若用户标注「待确认/请帮我推荐」→ 触发阶段 1.5
6. **目标市场**(可与目标国家不同;如多个市场细分) — 若用户标注「待确认/请帮我推荐」→ 触发阶段 1.5
7. **投放周期**(开始日期 + 结束日期)
8. **总预算**(金额 + 货币 + 日预算)

#### 4.A.4 WhatsApp 配置

WhatsApp 配置由宿主自动注入,不再列入用户必填字段:

- 宿主会自动注入 `waba_id` / `verified_name` / `display_number` / `phone_number_id` / `page_id`
- **场景 1:宿主注入 1 个号** → 直接使用,不打扰用户
- **场景 2:宿主注入 ≥2 个号** → 阶段 1.0 需求对接时简短询问:「您绑定了 N 个 WhatsApp 号(列出号码),本次投放用哪个?」
- **场景 3:宿主未注入** → 提示用户「请先到设置→WhatsApp Business 绑定后回来继续」

**严禁向用户询问 phone_number_id / page_id 等技术 ID**。

#### 4.A.5 选填字段(可标注 `[待客户补充]` 继续)

- 当地销售网络与维修设施
- 当地主要竞争对手(若不填则阶段二自动调研)
- 客户语气偏好(影响 welcome_message 措辞)
- 已有素材资产(产品图除外——产品图必须在阶段四开始前上传)

#### 4.A.6 阶段 1.0 内部审查(self-check,不展示给用户)

自检:必填字段全部收集或标注「待客户补充/待 1.5 推荐」、业务模式识别清晰、关键字段已提示用户补充。

#### 4.A.7 阶段 1.0 产出(在对话内输出)

- **需求清单**(markdown 格式,1 页内,所有字段一览)
- **业务模式简短描述**(2-3 行,告诉用户 AI 识别的方向)
- **遗漏项提示**(若有 `[待客户补充]` 字段)
- **决策辅助预告**(若有「待 1.5 推荐」字段,告诉用户进入 1.5 帮选)

输出后简要复述,询问是否调整,确认后:

- 若所有字段都已确认 → 跳到 §4.C 规则查询
- 若有「待 1.5 推荐」字段 → 进入阶段 1.5

---

### 4.B 阶段 1.5:决策辅助(仅在 1.0 有「待推荐」字段时触发)

#### 4.B.1 触发条件

阶段 1.0 中存在 ≥1 个字段被用户标记为「待确认 / 请帮我推荐」,涉及:

- **目标市场**(用户身份明确但市场不明)
- **目标车型**(品牌明确但车型不明)
- **目标国家**(车型市场明确但具体国家不明)

#### 4.B.2 决策辅助 ≠ 阶段二的市场分析

**阶段 1.5 决策辅助**:帮用户**做选择**(从未知到已知),输出推荐方案 + 用户确认,关键决策维度的数据足够即可。
**阶段 2 市场分析**:已知市场后**深度分析**(数据告诉用户怎么样),输出市场报告 + 数据洞察,全面深度。

**严格分工**:1.5 做完后,所确定的市场/车型/国家**回流到 1.0 的需求清单**,阶段 2 不再重复"帮选"。

#### 4.B.3 决策辅助执行流程

| Step | 动作 |
|---|---|
| 1 | 用户产品信息不全时(舵向/燃料类型/适用国家),AI 先询问用户上传/补充产品详细信息;用户提供后再做适配判断 |
| 2 | 用 web_search 拉一份候选清单(市场/车型/国家,根据用户待推荐字段) |
| 3 | 按关键决策维度评估:市场体量 / 竞争密度 / 政策友好度 / 产品适配度 |
| 4 | 输出 **Top1 + 备选 2** 推荐;若用户指定 N 个,就给 N 个 |
| 5 | 用户确认 → 回流到 1.0 需求清单的对应字段 |

**风险提示原则**:

- AI 仅做「**风险提示**」,**不替用户拍板**
- 例:「越南是右舵市场,您的左舵车进入存在适配风险,是否继续?」
- 不强制 web_search 校验某一项硬条件,不做硬筛闸门
- 让用户基于风险提示自己决策

#### 4.B.4 阶段 1.5 内部审查(self-check,不展示给用户)

自检:决策推荐基于真实数据(走 §5.0)、Top1+备选 2 给齐、风险提示到位、用户确认环节到位。

#### 4.B.5 阶段 1.5 产出(在对话内输出)

- **推荐方案**(Top1 + 备选 2,含决策依据简表)
- **风险提示**(若涉及)
- **用户确认请求**:「请确认推荐的 X,我会将其回填到需求清单」

用户确认后回流到 1.0 需求清单 → 进入 §4.C 规则查询。

### 4.C 平台 + 国家规则查询机制

**触发时机**:阶段 1.0 + 1.5 全部完成,目标平台 + 目标国家确定后,AI **自动**执行一次规则查询,**用户无感**。

#### 4.C.1 查询内容(用 web_search,基于物理时间)

| 查询类别 | 检索关键词模板 | 时效要求 |
|---|---|---|
| 目标平台广告规则 | `Meta CTW Click-to-WhatsApp ad policy {当前年月}` | < 3 个月 |
| 目标平台 welcome_message 规则 | `WhatsApp Business welcome message character limit format {当前年月}` | < 3 个月 |
| 目标国家广告法 | `{country} advertising law automotive ads {当前年月}` | < 3 个月 |

## 5. 阶段 2:市场分析(数据告诉用户「市场怎么样,优势在哪,往哪投」)

### 5.0 数据可信度三档制(贯穿阶段 2/3/5)

#### 5.0.1 数据三档定义(AI 内部使用)

| 档位 | 定义 | AI 处理方式 |
|---|---|---|
| 🟢 一手数据 | 官方报告/官网/政府公告/Meta Ads Library 直查 | web_search 拉取 + 内部记录 URL + 时间戳 |
| 🟡 二手推算 | 行业基准 + 当地修正系数,或多个间接来源交叉 | AI 内部标注"推算"二字 + 推算公式(内部使用) |
| 🔴 数据缺失 | 找不到任何可验证来源 | **禁止给具体数字**,改用定性描述,并在末尾提示用户"补充 XX 数据会让方案更准" |

**关键:三档标记 🟢🟡🔴 不展示给用户**,纯 AI 内部自检使用。

#### 5.0.2 「必须找」的判断标准

以下三类数据**必须**用 web_search 拉到一手来源,找不到就提示用户提供(上传/补充至知识库),不能用推算填充:

| 数据类型 | 举例 |
|---|---|
| **影响目标市场选择的** | 关税 / 外汇管制 / 驾驶方向 / Meta 可投性 / 目标国家广告法 |
| **影响 KPI 预测/预算分配的** | CPM 基准 / CTR 基准 / 行业平均 cost per conversation |
| **影响素材合规的** | 目标平台广告规范 / 目标国家广告法关键条款 |

#### 5.0.3 「不强求」的判断标准(可推算或缺失,不阻塞流程)

- 仅影响参考性内容(竞品打法描述、行业趋势叙述、case study 引用等)
- 这类数据找不到一手时,可标 🟡 推算或 🔴 缺失,在产出末尾提示用户"补充 XX 会让方案更准"

#### 5.0.4 数据缺失处理(分两类)

| 缺失类型 | 处理方式 |
|---|---|
| 「必须找」类缺失 | 输出"建议补充的数据清单",引导用户上传/加入知识库,**等用户补充后再继续**;若用户明确说"用现有数据先做一版",则继续但在末尾标注风险 |
| 「不强求」类缺失 | 用定性描述代替数字,末尾提示"补充 XX 数据会让方案更准" |

#### 5.0.5 时间基准动态规则

AI 每次 web_search 时,根据数据类型动态判断时效性要求:

| 数据类型 | 时效要求 | 举例 |
|---|---|---|
| **政策法规类** | < 3 个月 | 目标国家广告法、关税政策、外汇管制、Meta 广告政策更新 |
| **市场基准类** | < 12 个月 | CPM/CTR 行业基准、竞品 CTW 打法、市场规模、车型销量 |
| **历史背景类** | < 36 个月 | 行业历史趋势、车型发布信息 |

**时间基准 = 物理时间**(每次执行时取系统当前日期,**不是模型训练数据截止日**)。

**过期处理**:
- 数据时间超过对应阈值 → AI 内部标 ⚠️,重新 web_search 找最新
- 找不到更新的 → 在产出中用括号提示「数据时间:XXXX-XX,建议复核最新政策」

## 7. 阶段 4:CTW 素材生成(图片 + 文案 + welcome_message 文本)

### 7.1 开始前必须校验:用户上传的图片

`generate_ad_creative` 工具**强制要求** `reference_image_ids`(用户上传的产品图序号)。

#### 7.1.1 用户已上传图片 → AI 智能判断用途

AI 在阶段四调用 `generate_ad_creative` 前,先对会话中所有用户上传的图片做内部判断:

| 场景 | AI 处理方式 |
|---|---|
| 1 张图,内容明显是产品 | 直接用,不问 |
| 1 张图,内容是参考图/截图/对标图(非产品) | 不调用此图;若没有其他产品图,提示用户上传产品图 |
| 多张图,全部明显是产品 | 用全部图,不问 |
| 多张图,混合(产品图+非产品图) | AI 内部识别,只用产品图;若产品有多个车型/角度不明确,简短问一句「我看到您上传了 N 张图,请确认哪几张是 XX 车型的产品图」 |
| AI 完全无法判断 | 简短问一句:「请确认这张图是产品图吗?对应哪款车型?」 |

**原则**:AI 优先自己判断,只在不确定时才确认;一次性问完所有不确定项,**不每张图都问**。

#### 7.1.2 用户未上传图片 → 给出完整引导

```markdown
阶段四需要您先上传车型产品图。

**官方产品图获取路径**(按优先级):

1. 品牌方提供:授权代理商通常已有官方图库(联系品牌总部市场部索取)
2. 品牌官网 Media Kit:多数品牌官网底部有「图片资源 / Media / Press」页面
3. 品牌海外官方社媒:Facebook / Instagram 官方页面的高清产品图
4. 经销商培训资料:品牌发给经销商的培训手册中含官方图册

⚠️ **版权风险提示**:从网络获取图片会有版权风险,强烈建议本地上传授权的产品图。

**图片规格要求**:

- 透明背景 / 干净背景的官方图最佳
- 分辨率 ≥ 800×800
- 推荐角度:3/4 侧前角(最具广告吸引力)
- 每个车型 1-3 张高质量参考图就够了

请上传后告诉我图片对应哪款车型,我会基于您的产品图生成 Meta CTW 广告图。
```

**不要**用工具直接尝试,会返回 `{ error: 'no_reference_images' }` 后中断。

### 7.4 三步流程

**Step 1**:输出素材任务清单(表格列:素材 ID / 目标市场 / 调性 / 卖点焦点 / 场景描述要点)

**Step 2**:针对每个素材调用 `generate_ad_creative`,`product_description` 必须包含:

### 7.5 welcome_message 设计要求

每条 ad **必须**有 welcome_message,作为用户点击广告进入 WhatsApp 后看到的第一句话。

#### 7.5.1 基础要求

- 纯文本,**第一人称**(以品牌或销售身份说话)
- **必须包含产品名**(车型)
- **必须含一个开放式问题**(引导用户回复)
- 语言匹配目标市场(德国 → 德语 / 法国 → 法语 / 中东英语市场 → 英语)
- 经过母语者审核(产出物附 ⚠️ 审核提示)

#### 7.5.2 字符约束

**字符上限 = §4.C 锁定的目标平台规则中的 welcome_message 字段限制**(每次执行时由 web_search 拉取最新)。

- AI 内部从 `platform_country_rules` 字段读取当前平台限制
- 字符计数显示在 §7.6 的「素材汇总表」一列,不再写在 welcome_message 末尾
- AI 内部自检:每条 welcome_message 输出前对照已锁定的平台规则,超长立即重写

#### 7.5.3 业务模式分支

按阶段 1.0 锁定的业务模式倾向输出 welcome_message:

| 模式倾向 | welcome_message 口吻 |
|---|---|
| B2C 终端零售 | 终端消费者口吻,提试驾 / 到店 |
| B2B 跨境批发 | 经销商口吻,提 MOQ / FOB / OEM 支持 / 进口条件 |
| 品牌官方混合 | 综合口吻,先做询盘资质判定 |
| 自定义模式 | 按用户在 §4.A.2 的描述自定义 |

各市场各模式具体示例见 `read_skill_reference({ name: "meta-creative-specs" })` §2。

### 7.6 合规对照检查

每个素材的 headline / primary_text / welcome_message 生成后必须立即对照已锁定的规则(§4.C `platform_country_rules`)+ 通用黑名单 (`compliance-blacklist.md`),输出合规对照表:

```markdown
### 素材 CR-XXX 合规对照
| 字段 | 内容 | 黑名单触发 | 平台规则违反 | 处理 |
|---|---|---|---|---|
| headline | "..." | 无 / [禁词:XXX] | 无 / [超限:XXX] | ✅ 通过 / ❌ 重写为:... |
| primary_text | "..." | ... | ... | ... |
| welcome_message | "..." | ... | 字符 X/上限 Y | ✅ / ❌ |
```

**通用黑名单**(完整清单见 `read_skill_reference({ name: "compliance-blacklist" })`,7 类):绝对化用语 / 未求证对比(全禁) / 具体补贴金额 / 价格绝对值 + FOB 范围(全删,交给客服) / 性能绝对值 / 各市场附加禁词(动态查询)。

任何一项 ❌ 必须重写后才能交付。

### 7.7 阶段 4 内部审查(self-check,不展示给用户)

自检:素材合规对照表全部 ✅ 通过、welcome_message 字符约束符合 §4.C 锁定的平台规则、图片符合产品信息(用了正确车型的产品图,非参考图/截图)、业务模式倾向一致(welcome_message 口吻、文案调性符合阶段 1.0 识别的模式)、4 类失败模式约束在 prompt 中显式补齐。

审查不通过 → AI 内部重新生成或调整,通过后才输出。

### 7.8 阶段 4 产出(在对话内输出)

- **素材任务清单表**
- **每个素材**:url(由工具返回)+ Meta 文案 + welcome_message + 合规对照表(§7.6)
- **全素材汇总表**(列:素材 ID / headline / primary_text / welcome_message / 字符数(实际/上限,上限从 §4.C 读取)/ 合规通过 ✅❌)

输出后简要复述,询问是否调整。用户确认后才进入阶段 5。

详细规格调用 `read_skill_reference({ name: "meta-creative-specs" })`,合规黑名单调用 `read_skill_reference({ name: "compliance-blacklist" })`,素材 prompt 模板调用 `read_skill_reference({ name: "creative-prompt-patterns" })`。

## 8. 阶段 5:执行方案(全部 Meta 参数集中输出)

### 8.1 阶段 5 职责

阶段 5 是 Meta 具体参数的**唯一**集中地。

| 阶段 5 做什么 | 阶段 5 不做什么 |
|---|---|
| 输出 plan_json(全部 Meta 字段) | ❌ 不重复阶段 3 的策略论述 |
| 输出后台操作手册 | ❌ 不重复阶段 4 的素材内容(只引用 url) |
| 版位预算分配 | ❌ 不写阶段六(由宿主接续) |
| 受众配置(关键词,非 ID) | ❌ 不写 Pixel 落地页转化(锁定为 CAPI Conversation) |
| 出价策略 | |
| 字符约束按 §4.C 规则填(welcome_message 长度等) | |

### 8.2 双产物

阶段 5 产出包含**两个**:plan_json(严格 Meta API schema 的可执行 JSON)+ 后台操作手册(面向运营的手把手操作指引)。

### 8.3 plan_json 严格要求

- 字段值严格按 §2.1 锁定枚举使用
- 严禁出现 §2.2 禁用清单中的值
- `campaigns.length = 1`(多市场用同 campaign 下多个 ad_sets 按 countries 切分)
- `welcome_message` 字符长度按 §4.C 锁定的平台规则填,**不写死 300**
- WhatsApp 配置字段(`whatsapp_phone_number_id` / `page_id`)从宿主自动注入读取
- 每个 ad 必须有 welcome_message 文本(来自阶段 4)
- 每个 ad 必须有真实 image_url(来自阶段 4 generate_ad_creative 调用结果)

### 8.4 后台操作手册结构

详见 `read_skill_reference({ name: "meta-api-template" })`,核心步骤:前置准备 → Campaign 创建 → AdSet 创建 → Ad 创建 → WhatsApp 接入验证 → 启动前自检。

### 8.5 阶段 5 内部审查(self-check,不展示给用户)

AI 在交付 plan_json 前**最后一次**审查全链路结论一致性:

| 审查项 | 对照 |
|---|---|
| 目标市场 / 车型 / 国家 | = 阶段 1.0 + 1.5 锁定结论 |
| 受众 / 漏斗 / 创意方向 | = 阶段 3 锁定结论 |
| 素材内容(图片 + 文案 + welcome_message) | = 阶段 4 锁定结论 |
| 预算与周期 | = 阶段 1.0 输入 |
| 所有合规约束(字符 / 禁词 / 版位规则) | = §4.C 锁定的 `platform_country_rules` |
| Meta 字段枚举值 | = §2.1 锁定值,无 §2.2 禁用值 |
| WhatsApp 配置 | = 宿主自动注入值 |

任何一项不一致 → AI 内部自动调整,符合后才输出 plan_json。

### 8.6 阶段 5 产出(在对话内完整输出)

按以下顺序:

1. **plan_json**(完整 JSON,markdown 代码块包裹)
2. **后台操作手册**(完整 markdown)
3. **简短结论速览**(格式见 §3.5.1,字段含:总预算与日预算、目标市场与广告组数、素材数量、启动条件)

**不要写**「流程到此结束」「祝您投放顺利」等收尾语,宿主会接续调用 `draft_ad_plan`。

详见 `read_skill_reference({ name: "meta-api-template" })` 的 schema 与示例。

## 9. 全 Skill 执行原则

1. **CTW 锁定**:所有阶段产出涉及 Meta 投放参数时严格使用 §2.1 锁定值,清除 §2.2 禁用枚举
2. **阶段独立调用 + 依赖校验**:不强制顺序,但每阶段开始前按 §3.1 输入契约校验前置产出;走到 `draft_ad_plan` 仍需有真实 generate_ad_creative 调用记录
3. **对话内输出**:所有产出在 assistant 回复中以完整 markdown 直接呈现
4. **联网取数 + 时间基准**:阶段 1.5 / §4.C / 阶段 2 所有数据通过 `web_search` 获取最新值,时间基准按 §5.0.5 动态规则
5. **数据可信度三档**:每条数据按 §5.0.1 三档标注(AI 内部使用,**不展示给用户**)
6. **关键词不 ID**:targeting 字段输出关键词,不输出 Meta ID(skill 不能查 ID)
7. **阶段间锁定 + 修改往后传**:每阶段产出后,AI 内部维护 `latest_conclusion` 字段;用户修改只往后传不回改前置阶段
8. **修改交互**:1-2 个修改点直接输出整段更新;≥3 个修改点逐个改完后询问是否输出完整版
9. **业务模式单向禁忌**:严禁出现本模式的禁忌词;风格倾向为优先建议而非硬限制
10. **产品图前置**:阶段 4 开始前必须确认用户已上传**产品图**(非参考图/截图),AI 内部智能判断 + 必要时轻确认
11. **welcome_message 动态字符**:字符上限按 §4.C 锁定的平台规则,不写死 300
12. **合规对照**:阶段 4 每素材必过合规黑名单 + §4.C 平台/国家规则
13. **每阶段内部审查**:每阶段交付前 AI 自检,**审查日志不展示**,用户看到最终通过版
14. **WhatsApp 自动拉取**:由宿主注入,单号自动用,多号让用户在 1.0 阶段选,严禁问 phone_number_id
15. **不写阶段六**:阶段 5 结尾不收尾,宿主接续
