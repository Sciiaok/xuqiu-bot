## 一、改动综述

### 1.1 改动来源

skill 端业务侧反馈两项问题:

| 反馈 | 说明 |
|---|---|
| 输出过多 | 阶段三 17 章策划案对真实使用而言显著冗余,客户拿到一份 17 章 Word 体量的方案,真正能用上的字段只在其中 50%,其余是装饰性框架(SMART/SWOT/Brand Story 框架展开等),增加阅读成本 |
| 强制顺序执行不灵活 | 真实业务场景中存在三类用户:已做完市场分析想跳过阶段二、已有素材只缺 Meta 后台方案、只想做素材不要策略。强制全流程顺序导致这些用户必须从阶段一开始,效率低 |

经 skill 端与宿主端协商,达成两项契约调整。本次属于**契约的非破坏性微调**,但变更 2 涉及宿主 host-patch 行为,需要宿主端配合自查。

### 1.3 改动 2:五阶段顺序执行 → 独立调用

**变更前**(契约 v1.0 7.1 节):

> 五阶段顺序执行:需求对接 → 市场分析 → 策略生成 → 素材生成 → Meta 投放方案
>
> 每阶段开始前校验前置阶段产出完整性,缺失则提问补全

**变更后**(契约 v1.1 7.1 节):

> 五阶段框架(顺序仍为 一→二→三→四→五,作为依赖关系)
>
> **不强制顺序执行**,允许按用户场景从任意阶段进入。每阶段开始前按下表「输入契约」校验前置产出:
>
> - 缺失项较少 → 直接提问补全
> - 缺失项较多 → 走「快速通道」,用最简输入启动该阶段,缺失字段标 `[待客户补充]`

**关键守则**(避免独立调用变成失控):

| 项 | 维持 |
|---|---|
| 阶段间依赖关系 | 不变(阶段五依赖阶段三+四产出,阶段三依赖阶段一+二产出) |
| 用户产出确认机制 | 不变(每阶段产出后需用户确认才进入下一阶段) |
| 阶段四需用户上传产品图 | 不变(generate_ad_creative 工具强制要求) |
| 4.2 节 sanity check | 不变(走到 draft_ad_plan 仍必须有真实 generate_ad_creative 调用) |

## 三、宿主端配合事项(改动 2 专项)

### 3.1 host-patch 自查清单

请宿主集成方自查现行 `host-patch`(注入"阶段六:CTW 收口提交"指令的逻辑),回答以下问题:

| 问题 | 自查结果 | 是否需改动 |
|---|---|---|
| host-patch 是否假设"用户消息总是从阶段一开始"? | 是 / 否 | 是 → 需改动 |
| host-patch 是否在每轮对话都注入"阶段六"指令? | 是 / 否 | 是 → 需改为"仅在 skill 完成阶段五后注入" |
| host-patch 是否检查会话历史中是否有 generate_ad_creative 调用? | 是 / 否 | 否 → 建议增加(配合 sanity check) |
| host-patch 是否在用户跳过阶段时给出明确反馈? | 是 / 否 | 否 → 建议增加(优化用户体验) |

### 3.2 推荐的 host-patch 行为(参考实现思路)

```
当 skill 输出包含以下信号时,触发"阶段六:CTW 收口"指令注入:
  - 信号 1:输出包含 plan_json(完整 JSON 代码块)
  - 信号 2:输出包含"全量合规校验报告"
  - 信号 3:用户当前消息确认了阶段五产出(如"OK"、"确认"等)

不触发的场景:
  - skill 仅输出阶段一/二/三/四 产出
  - skill 主动声明"流程暂停"或"用户暂未确认"
```

### 3.3 4.2 节 sanity check 不动的理由

虽然变更 2 引入"独立调用",但 4.2 节的 sanity check **依然有效**——

| 场景 | 行为 |
|---|---|
| 用户从阶段一顺序跑到阶段五 | 会话历史中有 generate_ad_creative 调用,sanity 通过,正常调 draft_ad_plan |
| 用户跳过阶段四,直接让 skill 输出阶段五 plan_json | sanity 拦截,host-patch 无法成功调 draft_ad_plan,需要回头跑阶段四 |
| 用户只跑阶段三,不进入阶段五 | host-patch 不触发(无 plan_json),不调 draft_ad_plan,sanity 不被检查 |

这是正确的——sanity 的目的是"防止 image_url 造假",不是"强制顺序执行"。两者解耦后,独立调用 + sanity 共存毫无矛盾。

## 四、回归测试要求(上线前必做)

### 4.1 测试场景清单

| 场景编号 | 描述 | 期望行为 |
|---|---|---|
| **TC-01**(基线) | 用户从零开始,顺序跑 阶段一→二→三→四→五 | plan_json 正常生成,host-patch 注入阶段六,draft_ad_plan 调用成功 |
| **TC-02**(独立调用 - 阶段四) | 用户上传产品图,直接说"帮我生 5 张广告图,给一个简单的卖点描述" | skill 进入阶段四独立模式,要求最简必填(品牌/车型/卖点/语言/市场),调用 generate_ad_creative 生图,输出素材清单后**自然结束**(host-patch 不注入阶段六指令,因为没有 plan_json 信号) |
| **TC-03**(独立调用 - 阶段三) | 用户说"已经生过素材,需要一份 CTW 投放策划案" | skill 进入阶段三独立模式,要求最简必填+ 阶段二关键结论(可现场调研或用户提供),输出 10 章策划案后**自然结束** |
| **TC-04**(跳过阶段四,直接尝试阶段五) | 用户说"直接出 Meta 后台方案",未上传产品图 | skill 在阶段五开头校验输入契约,**拦截**并提示"需先生成素材(阶段四)";host-patch 不触发 |
| **TC-05**(无产品图但要阶段四) | 用户进入阶段四但未上传产品图 | skill 在调用 generate_ad_creative 前提示"请先上传产品图",不盲调工具 |
| **TC-06**(完整 brief 上传) | 用户上传 PDF brief,直接说"按这个跑全流程" | skill 解析 brief,只追问缺失项,然后顺序跑全流程 |

### 4.2 回归测试通过标准

- 6 个场景全部按"期望行为"列描述执行
- TC-04 必须拦截成功(sanity check 生效)
- TC-02、TC-03 不触发 draft_ad_plan(避免 host-patch 注入与独立调用语义冲突)
- TC-01 行为与 v1.0 完全一致

### 4.3 失败回滚预案

若任一场景失败:

1. 立即将 `skills/overseas-ad-planning.skill` 回滚到 v1.0(sha256 头 `375f242d`)
2. 联系 skill 作者排查问题
3. 修复后重新走回归测试
